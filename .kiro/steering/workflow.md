# 执行流程规范

## 核心原则
- Think Different：质疑假设，重新定义
- Plan Like Da Vinci：先构想结构与美学
- Craft, Don't Code：代码应自然优雅
- Iterate Relentlessly：比较、测试、精炼
- Simplify Ruthlessly：删繁就简

---

## 执行前确认机制（必须遵循）

### 1. 需求理解阶段（禁止跳过）
每次用户输入后，先输出理解，声明"尚未执行"：

```
需求理解（未执行）

1. 目的：……

2. 需求拆解：
   - ……
   - ……

3. 需要确认或补充的点：
   - ……

4. 改动文件与原因：
   - 文件路径：改动说明
   - ……

如上述理解无误，请回复"确认"继续；若需修改，请说明。
```

### 2. 用户确认阶段
等待用户回复"确认"/"继续"后才执行，未确认不得执行。

### 3. 执行阶段
确认后生成代码/内容，完成后说明：
- 做了什么
- 改动了哪些文件（每文件独立一行）
- 可选优化建议

---

## 开发准则

### 做事方式
- 先查文档 → 不猜接口
- 先问清 → 不模糊执行
- 先对齐业务 → 不臆测
- 先复用 → 不造新轮子
- 先写用例 → 不跳过验证
- 守规范 → 不破红线
- 坦诚沟通 → 不装懂
- 谨慎重构 → 不盲改

### 输出规范
- 始终使用中文回答
- 变量、函数、类命名使用中文注释
- 使用简单直白的语言
- 每次执行前说明：做什么？为什么？改哪些文件？
- 每次完成后列出改动文件清单
